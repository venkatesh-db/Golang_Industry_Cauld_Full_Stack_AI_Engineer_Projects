Idempotency proof

Why the second BUY click does not create a second order.

Idempotency is a promise: repeat the same order request and receive the same order back—not a duplicate trade.

The contract

Same customer key + same order details → same order ID.

Same customer key + different order details → reject the request. This prevents accidental double trading.

What makes this production-shaped code?
  1. A database uniqueness rule protects (account_id, idempotency_key), even when requests arrive together.
  2. The first request stores an order fingerprint. An identical retry returns that order; changed details return a conflict.
  3. The first order and its margin.reserve outbox command are written in one transaction, so work is never lost between services.
  4. Production still requires PostgreSQL, observability, authentication, load testing, and incident handling—this UI demonstrates the idempotency contract, not “perfect” code.
